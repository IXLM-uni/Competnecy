# Методология: Competency Pipeline

**Дата:** 2026-04-23
**Автор сессии:** совместно с Claude Opus 4.7
**Цель документа:** зафиксировать полный pipeline от сырого рыночного сигнала до ранжированной карты компетенций и syllabus-скелета программы обучения. Описать алгоритмы, метрики и ограничения — материал для публикации/диссертации.

---

## 1. Задача

По описанию роли (формулировка + контекст) автоматически построить:
1. Компетентностный корпус из рыночного сигнала (5 источников).
2. Единый каталог паспортов компетенций с дедупликацией и провенансом.
3. Ранжированную карту компетенций по важности для роли (5 tier-ов).
4. Syllabus-скелет программы обучения.

Решение об отсутствии консолидации: сохраняем все паспорта в полном объёме, включая Peripheral. Это даёт методисту полный рыночный сигнал и сохраняет провенанс каждой упоминаемой концепции.

Вложение в методологическую рамку Дубинского (CUSTIS/MODEUS): первая «долина смерти» (анализ деятельности → описание деятельности) автоматизирована, вторая (описание → learning outcomes) частично, третья (learning outcomes → активности) оставлена на следующий шаг.

Вложение в WGU CBE: наш пайплайн — открытая LLM-augmented альтернатива коммерческому Emsi/Lightcast. Заменяет маппинг Emsi на job postings на 5-источниковый LLM-mining.

## 2. Pipeline — 5 стадий

```
[Формулировка роли]
        │
        ▼
  Stage 1: Сбор корпусов (5 параллельных коллекторов)
        │
        ├─ hh.ru   (LLM select top-3 профессий → CrawlerClient)
        ├─ telegram (Qdrant RAG по tg_vacancy_channels)
        ├─ O*NET   (LLM select → SOC codes → XLSX)
        ├─ reddit  (whitelist 9 data-subreddits, спам-фильтр)
        └─ sonar   (Perplexity skill subprocess)
        │
        ▼
  Stage 2: Извлечение (2 LLM-прохода на источник)
        │
        ├─ EXTRACT  (источник-специфичный промпт, макс.покрытие)
        ├─ CLEAN    (бюджет удалений ≤25%, анти-галлюцинации)
        └─ META     (параллельно — контекст роли, не компетенции)
        │
        ▼ (5 профилей + 5 контекстов)
        │
  Stage 3: Merge 5 → 1 (LLM с 8 правилами + валидатор)
        │
        ▼ competency_passports.md (128 паспортов, 100% coverage, 0% hallucinations)
        │
  Stage 4: Ранжирование (2-сигнальный скоринг, батчинг)
        │
        ├─ Сигнал 1: частота источников (объективный, 0-5)
        └─ Сигнал 2: центральность для роли (LLM, 0-5)
        │
        ▼ competencies_ranked.md — ФИНАЛЬНЫЙ КАТАЛОГ (все 128 паспортов со Score/Tier)
        │
  Stage 5: Curriculum Skeleton
        │
        ▼ curriculum.md (модули, dependency, capstone — из Core+Important+Relevant)
```

**Примечание по консолидации.** Ранее был stage «консолидация Peripheral» (90 reddit-одиночек → 4 зонтичных паспорта, сжатие 128 → 43). Этот stage отключён: сохраняем все 128 паспортов в финальном каталоге, чтобы методист видел полный рыночный сигнал без потерь. Скрипт `consolidate_peripheral.py` оставлен в репозитории как опциональный инструмент, но не вызывается из `run.py --full`.

## 3. Источники — специфика и покрытие

### hh.ru (российский рынок, формализованные требования)
**Собираем:** LLM выбирает top-3 профессии из справочника hh.ru (полный CSV, ~400 профессий), CrawlerClient краулит listing pages и вакансии. 5 вакансий на профессию максимум.
**Сигнал:** требования работодателей, зарплаты, бенефиты, формат работы, описание обязанностей.
**Слепые зоны:** корпоративный жаргон, закрытые вакансии, обязанности описаны шаблонно.

### Telegram (городская неформальная рекрутинговая среда)
**Собираем:** Qdrant-коллекция `tg_vacancy_channels` (50K+ постов), embedding-поиск по role_scope.
**Сигнал:** зарплатные вилки (часто прямо в посте), контакты (@username), banковские кейсы (PD/LGD/LTV), позиции сверх hh.ru.
**Слепые зоны:** плохая структура, слипшийся текст.

### O*NET (официальная американская таксономия BLS)
**Собираем:** LLM выбирает top-3 occupations, XLSX даёт Skills/Knowledge/Core Tasks.
**Сигнал:** структурированные навыки с метриками Importance × Level, смежные роли, официальные определения.
**Слепые зоны:** англоязычный, нет русской банковской специфики, обновляется редко.

### Reddit (англоязычные обсуждения специалистов)
**Собираем:** whitelist из 9 тематических subreddits (r/datascience, r/analytics и т.д.), спам-фильтр (SLA Consultants, Udemy free deals).
**Сигнал:** тренды индустрии, эмоциональный фон, эволюция стека, AI-эра DS-ролей.
**Слепые зоны:** не даёт русских зарплат или работодателей. Comments endpoint без OAuth недоступен.

### Sonar (Perplexity-синтез через OpenRouter)
**Собираем:** 3 параллельных запроса (компетенции / обязанности / курсы) через skill-subprocess.
**Сигнал:** обзорный синтез, структура рабочего дня, образовательные программы.
**Слепые зоны:** обобщает (потеря конкретики в скобках), сноски `[1][2]` как шум.

## 4. Промпт-инженерия — ключевые технические решения

### 4.1. EXTRACT-промпт (Stage 2)
**Центральное правило:** «НЕ ОБОБЩАЙ, если в источнике есть конкретика». Для каждого источника — специфичные блоки, учащие LLM буквальности.
- `_SPEC_SONAR`: скобки — это перечисление, не группировка. «(BPMN, IDEF, UML)» = три отдельных термина.
- `_SPEC_TELEGRAM`: слитный текст, маркеры `—`, `·`, `🔹`. Названия позиций с англ. уточнением в скобках.
- `_SPEC_ONET`: таблицы Skills/Knowledge — построчное извлечение.
- `_SPEC_REDDIT`: строки-термины без глагольной формулировки («MongoDB», «MLOps») обязаны попасть как компетенции.
- **Объём:** для корпуса >15KB — ожидается 40-60 компетенций минимум.

### 4.2. CLEAN-промпт (Stage 2)
**Центральное правило:** «Если термина НЕТ в корпусе — удали». Анти-галлюцинационный проход.
**Анти-коллапс:** бюджет удалений ≤ 25% (иначе LLM слишком агрессивен). Английские vs русские написания — не галлюцинация.

### 4.3. MERGE-промпт (Stage 3)
**Структура выхода:** 4-поля + источники (formulation / tools / context / tasks / sources).
**8 правил:** все термины из корпуса / дедуп / ничего не пропускать / строки-термины в Инструменты / HH узкие формулировки / soft skills обязательны / провайдеры курсов игнорировать.
**Самопроверка:** Проход 1 (полнота) + Проход 2 (чистота), пока 0/0.

### 4.4. RANK-промпт (Stage 4)
**2 сигнала:**
- S1 = число источников (прямо из поля `Источники`, 0-5).
- S2 = LLM-оценка центральности (0-5) с иерархией: формулировка роли / ≥3 контекстов / 2 контекста / 1 контекст / косвенно / общая / противоречит.
**Батчинг:** 3 параллельных вызова по ≤45 паспортов каждый (обход Cloud.ru connection errors на больших выходах).
**Компактный выход + merge:** LLM даёт только `### formulation \n - Score: ... | Tier: ... | обоснование`, Python мёрджит в исходные паспорта.
**Автокоррекция Tier:** Score — источник истины; если LLM ошибся в границе tier, Tier пересчитывается.

### 4.5. CONSOLIDATE-промпт (опциональный, в основном пайплайне не используется)
Описан для полноты: схлопывает 90 «Знаком с концепцией X» (reddit-терминов-одиночек) в 4 зонтичных паспорта (Смежные роли / ML/DS стек / GenAI-LLM / MLOps-Cloud). В основном пайплайне отключён — финальным каталогом остаётся ранжированный список всех 128 паспортов.

## 5. Метрики (кейс: Аналитик данных в банке)

### Stage 2 → Stage 3 (merge)
- **Coverage 100%** — каждая bullet-строка из 5 профилей попала в каталог (валидатор с whitelist не-компетенций: провайдеры курсов, work-style descriptors, background-требования).
- **Hallucinations 0%** — все технические термины в паспортах буквально встречаются в корпусе.
- **Итерации до 100/0:** 3 (итеративная корректировка промпта).

### Stage 4 (ранжирование)
- Смаппилось 128/128 паспортов.
- Tier-mismatch (LLM ошибся в границе Score↔Tier): 25/128 = 19.5%, автоисправлено (довериться Score).
- Распределение: Core 7 / Important 3 / Relevant 5 / Specific 23 / Peripheral 90.
- Core 100% матчит формулировку роли: SQL, Excel, VBA, презентации, банк, коммуникация, дашборды.

### Финальный каталог
Сохраняем все 128 паспортов со Score/Tier. Консолидация не применяется по дизайнерскому решению: полный рыночный сигнал ценнее сжатия.

### Peripheral — состав (для справки)
90 peripheral-паспортов — почти все «Знаком с концепцией X» из reddit. Разбор по тематикам:
- 12 смежных ролей (Data Engineer, ML Engineer, quants, unicorn-role) — не компетенции.
- 23 ML/DS стек (Scikit-learn, PyTorch, BERT, Hadoop) — не для аналитика банка.
- 13 GenAI/LLM стек (OpenAI, RAG, LangChain) — не актуальны для роли.
- 11 MLOps/Infra (Docker, Kubernetes, AWS, SageMaker) — не актуальны.
- 7 шумовых (Oxylabs, HTML scraping).
- 5 дублей уже покрытых компетенций.

Вывод: **в Peripheral нет ложных-отрицательных** (важных компетенций не попало в низкий tier). Алгоритм 2-сигнального скоринга работает корректно.

## 7. Честные ограничения

Что пайплайн НЕ делает (по дизайну):

1. **Уровни освоения** (emerging → competent → proficient → mastery).
   Рыночный корпус их не содержит. Нужны либо syllabi курсов (решение отклонено: курсы врут и отстают от рынка), либо готовая рамка (Dreyfus / SFIA / Bloom) как ручная наложка, либо LLM-генерация (спекулятивно).

2. **Assessment-критерии** (performance / objective / preassessment rubrics).
   Нет данных для rubrics. Требует либо интервью-базы (Glassdoor, LeetCode), либо парсинг тестовых заданий, либо ручная работа методиста.

3. **Construct validity** (WGU research 2023: competency scores сильнее зависят от контекста оценивания, чем от самой компетенции).
   Мы не оцениваем студента — мы извлекаем требования. Но валидация нашей классификации tier пока только интерфейсная (ядро матчит формулировку роли).

4. **External validity** (связь extraction → успех на рынке).
   Не проверена. Для научной публикации нужно сравнить с резюме трудоустроенных специалистов или с тестовыми заданиями из реальных интервью.

## 9. Cross-role сравнение (Аналитик банка vs Backend Go)

Пайплайн прогнан на двух контрастных ролях для проверки устойчивости метода.

### Аналитик данных в банке
- **Stage 1+2:** 5 источников собраны полностью
- **Stage 3 (merge):** 128 паспортов, coverage 100%, hallucinations 0%
- **Stage 4 (rank):** Core 7 / Important 3 / Relevant 5 / Specific 23 / Peripheral 90 — ФИНАЛ
- **Stage 5 (curriculum):** 6 модулей, 16 компетенций в программе (из Core+Important+Relevant), capstone «Скоринг-дешборд с фрод-мониторингом»

### Backend разработчик на Go
- **Stage 1+2:** все 5 источников собраны (hh корпус сильно порезан CLEAN-этапом до 1.2KB — объяснение ниже)
- **Stage 3 (merge):** ~30 паспортов (меньше из-за сузившегося hh)
- **Stage 4 (rank):** Core 12 / Important 14 / Relevant 10 — ФИНАЛ (широкое ядро специализаций)
- **Stage 5 (curriculum):** 6 модулей, 36 компетенций, capstone «High-load Payment Gateway с сагами»

### Наблюдения

1. **Для более сфокусированной роли (аналитик банка в конкретной формулировке с инструментами xlsx/VBA/SQL) Core-tier меньше (7), и большинство паспортов уходит в Peripheral.** Роль имеет узкое ядро, остальное — знакомство с экосистемой.

2. **Для более широкой роли (Backend Go в распределённых системах) Core/Important/Relevant почти равные (12/14/10).** Нет одного ядра — каждая специализация нужна на средне-высоком уровне.

3. **Критическое замечание по Backend Go:** HH-профиль после CLEAN-этапа дал всего 1.2 KB. Это означает, что для Go на hh.ru либо мало вакансий в корпусе (LLM выбрал не те профессии из справочника), либо CLEAN слишком агрессивно удалил Go-специфичные термины (kubernetes/gRPC/Kafka и т.п.) как «не связанные с аналитикой». Требует отладки источник-специфичных промптов для не-аналитических ролей.

4. **Capstone-ы сгенерированы корректно** с учётом доменной специфики: для банка — фрод+скоринг, для Go — платёжный шлюз с сагами. Это подтверждает, что пайплайн правильно использует контекст роли из meta-файлов.

### Что из этого для научной публикации

- **Методология устойчива** к смене роли в плане структуры pipeline (все stage прошли без ручных правок).
- **Источник-специфичные промпты нуждаются в обобщении**: extract-правила для hh.ru сейчас заточены под аналитические роли («инструменты в скобках»), для инженерных ролей работают хуже.

## 10. Открытые направления для научной работы

1. **Масштабирование на роли**: прогнать пайплайн на 5-10 разных IT-ролях (Backend, Frontend, ML Engineer, Data Engineer, PM), сравнить распределения tier, найти cross-role core.
2. **Inter-source agreement как метрика валидности**: доказать эмпирически, что компетенции с частотой ≥3 источников предсказывают реальные требования к кандидату.
3. **A/B сравнение extraction-промптов**: какой промпт даёт лучше coverage / hallucinations / consolidation rate.
4. **External validity evaluation**: сравнить наш каталог с реальными резюме трудоустроенных аналитиков банков (N ≥ 100) и с тестовыми заданиями из интервью.
5. **Гибридная методология**: объединить наш рыночный мэппинг с Dreyfus-рамкой уровней для построения полноценной CBE-программы.
6. **Cross-lingual mapping**: добавить англоязычный рынок (LinkedIn Jobs, Indeed) и изучить divergence с русским (hh + telegram).
