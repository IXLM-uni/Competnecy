# Руководство к файлу LLM_SERVICE.md

## Архитектурная концепция: LLM-сервис как конструктор

### Предисловие: роль в общей архитектуре Use Case

`llm_service.py` реализует **сервисный слой** (Service Layer) в чистой архитектуре. Этот слой — **конструктор**, который проектирует и собирает ответ LLM из разнородных источников данных.

**Принцип:** Сервис НЕ содержит бизнес-логики сам по себе. Он предоставляет **блоки** (building blocks), которые бизнес-оркестраторы проектов-потребителей комбинируют под свои use cases.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         USE CASE LAYER (Проект-потребитель)                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │  Auth Flow   │  │ Chat Handler │  │ Doc Analysis │  │  Web Search  │     │
│  │  (Nea mini)  │  │  (MOVERS)    │  │  (VKMax)     │  │  (TGStat)    │     │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘     │
└─────────┼─────────────────┼─────────────────┼─────────────────┼─────────────┘

                    ┌───────────────▼───────────────┐
                    │   ChatOrchestrator (Service)  │
                    │     ───────────────────     │
                    │   Единая точка входа.       │
                    │   Собирает конвейер из      │
                    │   модульных компонентов     │

          │                         │                         │
┌─────────▼──────────┐  ┌──────────▼──────────┐  ┌──────────▼──────────┐

│   (всегда вместе)  │  │   (crawler4ai)      │  │   речи → текст     │

                    │   • Streaming                 │

                    │   • JSON structured output    │
                    └───────────────────────────────┘
```

### Кирпичики системы (Building Blocks)

| Блок | Компоненты | Суть |
|------|-----------|------|
| **RAG** | `Retriever` + `QdrantVectorStore` + `Dense/Sparse EmbeddingClient` | Гибридный поиск: dense-семантика + sparse-лексика всегда вместе через RRF/DBSF fusion |
| **Crawler** | `CrawlerClient` + `CrawlerQueryRewriter` | Интернет-поиск через DuckDuckGo → краулинг контента |
| **ASR** | `ASRClient` | Аудио → текст через Cloud.ru Whisper |
| **Документы** | `DocumentIngestor` + `Chunker` | Парсинг 6+ форматов → чанкинг с перекрытием |
| **Semantic Scholar** | `S2Client` + `S2SearchFilter` + `S2FieldInference` | Поиск статей, авторов, цитирований, рекомендаций + LLM-инференс fieldsOfStudy |

### Паттерн "конструктор"

```python
# Проект-потребитель собирает оркестратор под свой use case:
orchestrator = ChatOrchestrator(
    context_builder=ContextBuilder(),
    llm_client=OpenAIClient(),
    retriever=Retriever(embedding_client, vector_store, sparse_client),  # RAG
    crawler_client=CrawlerClient(),                                       # Internet
    asr_client=ASRClient(),                                               # ASR
    document_ingestor=DocumentIngestor(),                               # Docs
    tool_registry=ToolRegistry(),                                         # Tools
)

# Затем использует:
result = await orchestrator.handle_user_input(user_input, ctx)
```

### Контракты между слоями

| Контракт | Назначение | Ключевые поля |
|----------|-----------|---------------|
| `UserInput` | Вход пользователя | `text`, `audio_ref`, `files[]`, `mode`, `use_internet`, `enable_tools`, `enable_asr` |
| `RequestContext` | Контекст запроса | `request_id`, `user_id`, `rag_filters`, `tool_call_limit` |
| `OrchestratorResult` | Выход оркестратора | `response_text`, `sources[]`, `tool_calls[]`, `tokens_used` |
| `StreamEvent` | Стриминговые события | `kind`, `step`, `data` |

## UC-1: Пакетная индексация документов в RAG (Batch Ingestion)

**Prerequisites:**
- **Аутентификация:** Запрос выполняется от имени администратора через `Authorization` header с Bearer-токеном
- **Payload:** `{"files": [...], "collection_name": "documents", "metadata": {...}}`
- **Доступ:** Документы индексируются в указанную коллекцию Qdrant и доступны для RAG-поиска всем пользователям с правами доступа к этой коллекции
- **Коллекция:** `documents` (или проектно-специфичная, например `vkmax_docs`, `movers_knowledge`)

**Actor:** Администратор системы / Фоновый процесс

**Цель:** Массовая загрузка и индексация документов в векторное хранилище.

**Шаги реализации:**

1. **ШАГ 1. Получение списка файлов.**
   - Функция: `get_files_batch(directory_path)` или API endpoint
   - Валидация: проверка MIME-типов (PDF, DOCX, TXT, MD, HTML)
   - Логирование: «ШАГ 1. Получили N файлов для индексации — УСПЕХ»

2. **ШАГ 2. Параллельный инжест.**
   - Функция: `DocumentIngestor.ingest(file, ctx)` для каждого файла
   - Ограничение конкурентности через `asyncio.Semaphore(5)`
   - Логирование: «ШАГ 2. Инжест файла X из N — ОТПРАВЛЯЕМ / УСПЕХ / ОШИБКА»

3. **ШАГ 3. Чанкинг.**
   - Функция: `Chunker.split(text, source_id)`
   - Параметры: chunk_size=512, overlap=128
   - Логирование: «ШАГ 3. Файл X разбит на Y чанков — УСПЕХ»

4. **ШАГ 4. Генерация эмбеддингов.**
   - Функция: `CloudRuEmbeddingClient.embed_batch(chunks)` — dense векторы
   - Функция: `SparseEmbeddingClient.embed_batch(chunks)` — sparse векторы
   - Логирование: «ШАГ 4. Эмбеддинги для файла X — УСПЕХ»

5. **ШАГ 5. Индексация в Qdrant.**
   - Функция: `QdrantVectorStore.upsert_points(points, metadata)`
   - Метаданные: `IndexMetadata(document_name, user_id, tags, custom_data)`
   - Логирование: «ШАГ 5. Индексация файла X — УСПЕХ»

6. **ШАГ 6. Отчет.**
   - Функция: формирование JSON-отчета с результатами
   - Возврат: `{processed: N, failed: M, total_chunks: K}`
   - Логирование: «ШАГ 6. Пакетная индексация завершена — УСПЕХ»

## UC-2: Мультимодальный запрос (аудио + документы + текст)

**Prerequisites:**
- **Payload вариант A (текстовый query):**
  ```json
  {
    "query": "текстовый запрос или транскрипт аудио",
    "collection_name": "documents",
    "rag_filters": {"tags": ["category"], "user_id": "..."},
    "files": ["file_id_1", "file_id_2"],
    "audio_ref": "audio_file_id",
    "enable_asr": true,
    "use_internet": false,
    "mode": "rag_tool"
  }
  ```
- **Payload вариант B (query = аудиофайл, crawler отключен):**
  ```json
  {
    "query": null,
    "audio_ref": "audio_file_id",
    "enable_asr": true,
    "use_internet": false,
    "files": ["file_id_1"],
    "collection_name": "documents",
    "mode": "rag_tool"
  }
  ```
  - При `query=null` и `enable_asr=true` — текст запроса берется из транскрипции аудио
  - При `use_internet=false` — краулер не используется, только RAG по коллекции
- **Коллекция:** Указывается явно в `collection_name` для RAG-поиска (например `"documents"`, `"knowledge_base"`)
- **Требования:** `ASRClient` сконфигурирован (`ASR_API_KEY` в .env)

**Actor:** Пользователь

**Цель:** Комбинированный запрос с несколькими источниками данных.

**Шаги реализации:**

1. **ШАГ 1. Прием комплексного запроса.**
   - Пользователь отправляет: аудио-вопрос + PDF-документ + текстовое уточнение
   - Функция: формирование `UserInput` с `audio_ref`, `files[]`, `text`
   - Флаги: `enable_asr=true`, `mode="rag_tool"`
   - Логирование: «ШАГ 1. Получили мультимодальный запрос — УСПЕХ»

2. **ШАГ 2. ASR + Инжест параллельно.**
   - Функции (asyncio.gather):
     - `ASRClient.transcribe(audio_ref)` — распознавание голоса
     - `DocumentIngestor.ingest(file, ctx)` — извлечение текста из PDF
   - Логирование: «ШАГ 2. ASR и инжест документа — ОТПРАВЛЯЕМ»

3. **ШАГ 3. Объединение входных данных.**
   - Функция: `ContextBuilder.build_input()`
   - Контекст включает: транскрипт аудио + чанки документа + текстовое уточнение
   - Логирование: «ШАГ 3. Контекст собран — УСПЕХ»

4. **ШАГ 4. RAG-поиск (опционально).**
   - Если нужна внешняя информация: `Retriever.search(rewritten_query)`
   - Гибридный поиск: dense + sparse + RRF fusion
   - Логирование: «ШАГ 4. RAG-поиск — УСПЕХ»

5. **ШАГ 5. LLM-ответ.**
   - Функция: `ChatOrchestrator.stream_user_input()`
   - Стриминг токенов с учетом всех источников
   - Логирование: «ШАГ 5. Генерация ответа — УСПЕХ»

6. **ШАГ 6. Финальный ответ.**
   - Возврат: структурированный ответ с ссылками на источники
   - Логирование: «ШАГ 6. Мультимодальный запрос обработан — УСПЕХ»

## UC-3: Поиск информации в интернете (Web Search + Crawler)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "query": "поисковый запрос",
    "use_internet": true,
    "max_results": 5,
    "word_count_threshold": 100,
    "collection_name": null
  }
  ```
- **Конфигурация:** `CrawlerClient` доступен и сконфигурирован (`CRAWLER_BASE_URL` в .env)
- **Внешние зависимости:** Доступ к DuckDuckGo для поиска, Crawler4AI-сервер для краулинга
- **Ограничения:** Лимит на количество страниц (`max_pages`), минимальное количество слов (`word_count_threshold`)

**Actor:** Пользователь / Исследователь

**Цель:** Поиск актуальной информации в интернете через DuckDuckGo + краулинг найденных страниц.

**Шаги реализации:**

1. **ШАГ 1. Получение поискового запроса.**
   - Пользователь отправляет: `query` + `use_internet=true`
   - Валидация: проверка длины запроса (минимум 3 символа)
   - Логирование: «ШАГ 1. Получен поисковый запрос — УСПЕХ»

2. **ШАГ 2. Поиск через DuckDuckGo.**
   - Функция: `CrawlerQueryRewriter.rewrite(query, n=2)` — генерация вариантов запроса
   - Функция: `CrawlerClient.search(rewritten_queries, max_results)`
   - Возврат: список URL с релевантными страницами
   - Логирование: «ШАГ 2. DuckDuckGo вернул N результатов — УСПЕХ»

3. **ШАГ 3. Параллельный краулинг URL.**
   - Функция: `CrawlerClient.crawl_urls(urls, ctx)` для каждого URL
   - Ограничение конкурентности: `asyncio.Semaphore(5)`
   - Параметры: `word_count_threshold=100` (фильтр коротких страниц)
   - Логирование: «ШАГ 3. Краулинг %d URL — ОТПРАВЛЯЕМ / УСПЕХ / ОШИБКА»

4. **ШАГ 4. Извлечение и чанкинг контента.**
   - Функция: `Chunker.split(crawled_text, source_id=URL)`
   - Фильтрация: удаление навигации, рекламы, футеров
   - Логирование: «ШАГ 4. Контент разбит на N чанков — УСПЕХ»

5. **ШАГ 5. RAG-поиск по внешнему контенту (опционально).**
   - Если `collection_name` указана: индексация чанков во временную коллекцию
   - Функция: `Retriever.search(query, collection=temp_collection)`
   - Логирование: «ШАГ 5. RAG по внешнему контенту — УСПЕХ»

6. **ШАГ 6. LLM-ответ с источниками.**
   - Функция: `ContextBuilder.build_input()` с сниппетами из crawled-контента
   - В ответе указываются источники: URL + заголовок страницы
   - Логирование: «ШАГ 6. Ответ с веб-источниками отправлен — УСПЕХ»

## UC-4: Автоматическое суммирование документов (Map-Reduce)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "files": ["file_id_1", "file_id_2"],
    "summary_type": "short",
    "max_length": 500,
    "output_language": "ru",
    "map_reduce": true
  }
  ```
- **Форматы:** PDF, DOCX, TXT, MD, HTML (MIME-типы проверяются)
- **Классы из `llm_service.py`:**
  - `DocumentIngestor` — инжест документов
  - `Chunker` — разбиение на чанки
  - `OpenAIClient` — запросы к LLM
  - `JsonRepairLLM`, `StrictOutputParser` — валидация JSON-ответов
  - `RequestContext` — контекст запроса
  - `FileRef` — ссылка на файл
- **Ограничения:** Максимальный размер файла — 40 МБ

**Actor:** Пользователь / Менеджер / Автоматизированный процесс

**Цель:** Создание краткого summary документов с использованием Map-Reduce подхода для больших документов.

**Шаги реализации:**

1. **ШАГ 1. Загрузка и инжест документа.**
   - Функция: `DocumentIngestor.ingest(file, ctx)`
   - Извлечение текста из PDF/DOCX/TXT/MD/HTML
   - Логирование: «ШАГ 1. Документ получен — УСПЕХ»

2. **ШАГ 2. Чанкинг документа.**
   - Функция: `Chunker.split(text, source_id)`
   - Параметры: `chunk_size_tokens=512`, `chunk_overlap_tokens=128`
   - При небольшом документе (< 1 чанка) — прямое суммирование без Map-Reduce
   - Логирование: «ШАГ 2. Документ разбит на N чанков — УСПЕХ»

3. **ШАГ 3. Map-фаза: параллельное суммирование чанков.**
   - Для каждого чанка: запрос к LLM через `OpenAIClient.complete()`
   - Промпт: краткое резюме фрагмента (3-5 предложений)
   - Ограничение конкурентности: `asyncio.Semaphore(3)`
   - Логирование: «ШАГ 3. Суммирование чанка X из Y — ОТПРАВЛЯЕМ / УСПЕХ»

4. **ШАГ 4. Reduce-фаза: агрегация partial summaries.**
   - Объединение partial summaries в один текст
   - Если агрегированный текст слишком длинный — рекурсивный Reduce с повторным чанкингом
   - Логирование: «ШАГ 4. Агрегация partial summaries — УСПЕХ»

5. **ШАГ 5. Финальное суммирование.**
   - Функция: `OpenAIClient.complete()` с финальным промптом
   - Типы summary: `short` (3-5 предложений), `detailed` (10-15), `bullet_points` (5-7 пунктов)
   - Парсинг ответа через `StrictOutputParser.parse_json()` при необходимости
   - Логирование: «ШАГ 5. Финальное summary — УСПЕХ»

6. **ШАГ 6. Извлечение ключевых моментов.**
   - Функция: `OpenAIClient.complete()` с промптом на извлечение ключевых пунктов
   - Формат: JSON-массив строк
   - Парсинг: `StrictOutputParser.parse_json()` с ремонтом через `JsonRepairLLM.repair()`
   - Логирование: «ШАГ 6. Ключевые моменты извлечены — УСПЕХ»

7. **ШАГ 7. Возврат результата.**
   - Возврат структурированного ответа с полями: `summary`, `key_points`, `document_info`
   - Логирование: «ШАГ 7. Summary создано — УСПЕХ»

**Оптимизации:**
- **Конкурентность Map-фазы:** `asyncio.Semaphore(3)` — не более 3 параллельных запросов к LLM
- **Рекурсивный Reduce:** При превышении лимита токенов — повторный чанкинг и Reduce
- **Кеширование:** Результаты суммирования можно кешировать по `file.checksum`

## UC-5: Генерация структурированных данных (JSON Schema)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "user_query": "Текст запроса пользователя (обязательно)",
    "response_schema": {
      "type": "object",
      "properties": {"name": {"type": "string"}, "age": {"type": "number"}},
      "required": ["name", "age"]
    },
    "context": "дополнительный контекст (опционально)"
  }
  ```
- **Требование:** `user_query` обязателен — это запрос пользователя, на который LLM должен ответить в соответствии со схемой

**Actor:** Разработчик / API-клиент

**Цель:** Получение ответа LLM в строго заданном JSON-формате.

**Шаги реализации:**

1. **ШАГ 1. Определение схемы.**
   - Клиент передает `response_schema` (JSON Schema)
   - Пример: `{name: string, age: number, items: array}`
   - Логирование: «ШАГ 1. Получена схема ответа — УСПЕХ»

2. **ШАГ 2. Подготовка запроса.**
   - Функция: `ContextBuilder.build_input()` с инструкцией
   - Инструкция: «Ответь строго в формате JSON по заданной схеме»
   - Логирование: «ШАГ 2. Промпт с инструкцией готов — УСПЕХ»

3. **ШАГ 3. LLM-запрос с constrained output.**
   - Функция: `OpenAIClient.complete(request)`
   - Параметр: `response_format={"type": "json_object"}`
   - Логирование: «ШАГ 3. Запрос к LLM — ОТПРАВЛЯЕМ»

4. **ШАГ 4. Валидация и ремонт JSON.**
   - Функция: `StrictOutputParser.parse_json(text, schema)`
   - При ошибке: `JsonRepairLLM.repair(raw_text)` + повторная валидация
   - Логирование: «ШАГ 4. Валидация JSON — УСПЕХ / РЕМОНТ / ОШИБКА»

5. **ШАГ 5. Возврат валидированного результата.**
   - Возврат: распарсенный JSON, прошедший валидацию Pydantic
   - При невозможности ремонта: возврат ошибки с деталями
   - Логирование: «ШАГ 5. Структурированный ответ отправлен — УСПЕХ»

## UC-6: RAG-поиск с динамическими фильтрами по метаданным

**Prerequisites:**
- **Коллекция:** Явно указывается в `RequestContext.collection_name` (например `"documents"`, `"vkmax_knowledge"`)
- **Payload:**
  ```json
  {
    "query": "поисковый запрос",
    "collection_name": "documents",
    "filters": {"user_id": "123", "tags": ["finance"], "date_from": "2024-01-01"},
    "limit": 10
  }
  ```

**Actor:** Пользователь / Исследователь

## UC-7: Асинхронная обработка долгих задач с callback (AsyncTaskQueue)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "user_input": {...},
    "callback_url": "https://client.com/webhook",
    "webhook_secret": "secret_key_for_signature",
    "task_timeout": 3600,
    "max_retries": 3
  }
  ```
- **Конфигурация:** `AsyncTaskQueue` инициализирована в `ChatOrchestrator`
- **Внешние зависимости:** `callback_url` должен быть доступен из сети сервиса
- **Требования:** Клиент должен принимать POST-запросы с JSON-payload и верифицировать подпись по `webhook_secret`

**Actor:** API-клиент / Внешняя система

**Цель:** Выполнение тяжелых операций (большие документы, длинные цепочки, batch-обработка) без блокировки HTTP-запроса.

**Основные компоненты (реализованы локально в `UC_7.py`):**
- `AsyncTask` — модель асинхронной задачи со статусами: pending, processing, completed, failed
- `AsyncTaskQueue` — очередь задач с worker pool (asyncio.Queue + N параллельных workers)
- `WebhookClient` — отправка callback с подписью HMAC-SHA256

**Шаги реализации:**

1. **ШАГ 1. Прием асинхронного запроса.**
   - Клиент отправляет: `UserInput` + `callback_url` + `webhook_secret`
   - API endpoint создаёт `AsyncTask` со статусом `pending`
   - Функция: `task_queue.enqueue(task)`
   - Логирование: «ШАГ 1. Создана асинхронная задача %s — УСПЕХ»

2. **ШАГ 2. Постановка в очередь.**
   - `AsyncTaskQueue` сохраняет задачу в `asyncio.Queue`
   - При наличии persistence — сохранение в Redis/DB
   - Возврат клиенту: `{task_id, status: pending, estimated_time: "~2 min"}`
   - Логирование: «ШАГ 2. Задача %s поставлена в очередь — УСПЕХ»

3. **ШАГ 3. Взятие в работу worker'ом.**
   - Свободный worker берёт задачу из очереди
   - Статус меняется на `processing`, `started_at = now()`
   - Логирование: «ШАГ 3. Задача %s взята в работу worker %d — УСПЕХ»

4. **ШАГ 4. Фоновая обработка.**
   - Worker вызывает: `ChatOrchestrator.handle_user_input()`
   - При длительных операциях обновление `task.progress_percent`
   - Логирование: «ШАГ 4. Обработка задачи %s — В ПРОЦЕССЕ (прогресс %d%%)»

5. **ШАГ 5. Завершение и callback.**
   - По окончании: `status=completed`, сохранение `result`
   - Функция: `WebhookClient.send(task)`
   - Payload с подписью HMAC (если указан `webhook_secret`)
   - Логирование: «ШАГ 5. Callback для задачи %s отправлен — УСПЕХ»

6. **ШАГ 6. Обработка ошибок и retry.**
   - При ошибке: инкремент `current_retry`, проверка `max_retries`
   - Если остались попытки — возврат в очередь с задержкой (exponential backoff)
   - Если исчерпаны — `status=failed`, отправка callback с ошибкой
   - Логирование: «ШАГ 6. Задача %s — ОШИБКА, retry %d/%d» или «FAILED»

## UC-10: Классификация документов через Qdrant (Payload-based k-NN Classification)

**Prerequisites:**
- **Коллекция:** `documents` (или другая существующая коллекция с полем `category` в payload)
- **Payload при индексации:**
  ```json
  {
    "text": "содержимое документа",
    "metadata": {
      "category": "legal",
      "subcategory": "contract",
      "confidence": 0.95
    }
  }
  ```
- **Требования:** Коллекция должна содержать достаточное количество документов с известными категориями (минимум 5 на категорию)
- **Конфигурация:** `CloudRuEmbeddingClient` для генерации эмбеддингов
- **Как работает:** k-NN classification — поиск k ближайших соседей с известной категорией + majority voting

**Actor:** Администратор / Автоматизированный процесс

**Цель:** Автоматическое определение категории документа через поиск похожих в Qdrant и присвоение метки в payload.

**Шаги реализации (полный flow):**

1. **ШАГ 1. Загрузка документа.**
   - Система получает документ через API upload
   - Функция: `DocumentIngestor.ingest(file, ctx)`
   - Логирование: «ШАГ 1. Документ для классификации получен — УСПЕХ»

2. **ШАГ 2. Генерация эмбеддинга.**
   - Функция: `CloudRuEmbeddingClient.embed(document_text)`
   - Получаем dense-вектор для поиска похожих
   - Логирование: «ШАГ 2. Эмбеддинг документа получен — УСПЕХ»

3. **ШАГ 3. Поиск ближайших соседей с категориями.**
   - Функция: `QdrantVectorStore.search_with_filter(embedding, filter={"exists": "category"}, limit=5)`
   - Находим 5 похожих документов, у которых уже есть категория в payload
   - Логирование: «ШАГ 3. Найдено %d похожих документов с категориями — УСПЕХ»

4. **ШАГ 4. Голосование по категориям (k-NN classification).**
   - Извлекаем категории из payload найденных документов
   - Алгоритм: majority voting (выбираем категорию большинства соседей)
   - Вычисляем confidence: доля соседей с выбранной категорией
   - Логирование: «ШАГ 4. Предсказана категория '%s' с уверенностью %.2f — УСПЕХ»

5. **ШАГ 5. Сохранение документа с категорией.**
   - Функция: `QdrantVectorStore.upsert()` с payload содержащим `category`, `confidence`, `classified_at`
   - Документ индексируется в коллекцию с присвоенной меткой
   - Логирование: «ШАГ 5. Документ индексирован с категорией — УСПЕХ»

6. **ШАГ 6. Фильтрация по категориям при поиске.**
   - Теперь можно искать только по документам категории:
     ```python
     filter = {"must": [{"key": "category", "match": {"value": "legal"}}]}
     ```
   - Логирование: «ШАГ 6. Категория доступна для фильтрации в RAG — УСПЕХ»

   - Логирование: «ШАГ 2. Транскрипция файла X из N — ОТПРАВЛЯЕМ / УСПЕХ / ОШИБКА»

3. **ШАГ 3. Сбор результатов.**
   - `asyncio.gather()` для параллельного выполнения всех задач
   - Формирование `SingleTranscriptResult` для каждого файла
   - Подсчёт статистики: success_count, error_count, total_duration
   - Логирование: «ШАГ 3. Собраны результаты транскрипции — УСПЕХ»

4. **ШАГ 4. Пост-обработка (опционально).**
   - Если `enable_post_processing=true`:
     - Суммирование транскриптов через `DocumentSummarizer`
     - Извлечение ключевых слов
   - Логирование: «ШАГ 4. Пост-обработка — УСПЕХ»

5. **ШАГ 5. Формирование отчёта.**
   - Создание `BatchTranscriptResult`
   - JSON-структура:
     ```json
     {
       "batch_id": "uuid",
       "total_files": 10,
       "success_count": 9,
       "error_count": 1,
       "results": [
         {
           "file_path": "/path/to/file.mp3",
           "status": "success",
           "transcript": {"text": "...", "segments": [...]},
           "processing_time": 12.5
         }
       ]
     }
     ```
   - Логирование: «ШАГ 5. Отчёт сформирован — УСПЕХ»

6. **ШАГ 6. Callback (опционально).**
   - Если указан `callback_url` — отправка через `WebhookClient`
   - Payload содержит `batch_id`, статистику и результаты
   - Логирование: «ШАГ 6. Callback отправлен — УСПЕХ»


**Оптимизации:**
- **Семафор конкурентности:** `asyncio.Semaphore(3)` — не более 3 параллельных запросов к Cloud.ru ASR
- **Индивидуальная обработка:** Каждый файл обрабатывается отдельно, ошибка одного не влияет на другие
- **Детальный отчёт:** Полная информация по каждому файлу для диагностики
- **Возобновляемость:** При сбое можно повторно обработать только файлы с ошибками

## UC-15: Deep Academic Research (Semantic Scholar → RAG → LLM)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "query": "Как трансформеры изменили обработку естественного языка?",
    "search_queries_count": 3,
    "papers_per_query": 10,
    "top_k_relevant": 5,
    "collection_name": "s2_research_<hash>",
    "output_language": "ru"
  }
  ```
- **Внешние зависимости:** `pip install semanticscholar` (библиотека `semanticscholar`)
- **API:** Semantic Scholar Academic Graph API (`/graph/v1/paper/search`)
- **Конфигурация:** `CLOUDRU_API_KEY`, `QDRANT_URL` в .env; `S2_API_KEY` (опционально, для повышенных лимитов)
- **Ограничения:** Rate limit S2 API: 1 req/sec (без ключа), 10 req/sec (с ключом)

**Actor:** Исследователь / Аналитик / Пользователь

**Цель:** Получить глубокий, подкреплённый научными источниками ответ на произвольный вопрос. Полный pipeline: генерация поисковых запросов → S2FieldInference (авто-определение fieldsOfStudy) → S2Client.search_papers с фильтрами → LLM-субагент отбирает 5 релевантных → чанкинг + эмбеддинг в RAG-коллекцию → гибридный поиск → LLM-ответ с цитированием.

**Шаги реализации:**

1. **ШАГ 1. Генерация поисковых запросов через LLM.**
   - Функция: `OpenAIClient.complete()` с промптом: «Сгенерируй 3 разных поисковых запроса на английском для Semantic Scholar по теме: [query]»
   - Формат ответа: JSON-массив строк
   - Парсинг: `StrictOutputParser.parse_json()` + `JsonRepairLLM.repair()`
   - Логирование: «ШАГ 1. Сгенерировано %d поисковых запросов — УСПЕХ»

1b. **ШАГ 1b. LLM-инференс fieldsOfStudy.**
   - Функция: `S2Client.infer_fields(query, ctx)` — автоматическое определение научных областей через LLM
   - Формирует `S2SearchFilter(fields_of_study=[...])` для фильтрации поиска
   - Логирование: «ШАГ 1b. Инференс fieldsOfStudy: [...] — УСПЕХ»

2. **ШАГ 2. Поиск статей в Semantic Scholar (30 штук).**
   - Функция: `S2Client.search_papers(query, limit=10, filters=S2SearchFilter)` × 3 запроса
   - Поля: `paperId, title, abstract, year, citationCount, authors, openAccessPdf, tldr, fieldsOfStudy`
   - Дедупликация по `paperId`, фильтрация по fieldsOfStudy
   - Ограничение: `S2_RATE_LIMIT_DELAY = 0.6с` между запросами
   - Логирование: «ШАГ 2. Найдено %d уникальных статей из %d запросов — УСПЕХ»

3. **ШАГ 3. LLM-субагент отбирает 5 релевантных статей.**
   - Функция: `OpenAIClient.complete()` — отдельный вызов с минимальным контекстом (без истории)
   - Промпт: «Из 30 статей выбери 5 наиболее релевантных запросу [query]. Верни JSON с paper_ids и reasoning»
   - Входные данные: title + abstract (сокращённый до 200 символов) для каждой статьи
   - Парсинг: `StrictOutputParser.parse_json()`
   - Логирование: «ШАГ 3. LLM-субагент выбрал %d статей из %d — УСПЕХ»

4. **ШАГ 4. Формирование текстов выбранных статей.**
   - Функция: `S2Client.paper_to_text(paper)` — статический метод формирования текстового представления
   - Составление расширенного текста: title + authors + year + abstract + tldr
   - Логирование: «ШАГ 4. Загружены полные данные %d статей — УСПЕХ»

5. **ШАГ 5. Чанкинг + эмбеддинг + индексация в Qdrant.**
   - Функция: `Chunker.split(paper_text, source_id=paper_id)` для каждой статьи
   - Функция: `ingest_and_index()` — dense + sparse эмбеддинги → upsert в коллекцию `s2_research_<hash>`
   - Метаданные: `IndexMetadata(document_name=title, tags=["s2", year], custom_data={"paper_id": ..., "citation_count": ...})`
   - Коллекция создаётся временная под запрос (TTL или ручное удаление)
   - Логирование: «ШАГ 5. Индексировано %d чанков из %d статей — УСПЕХ»

6. **ШАГ 6. RAG-поиск по коллекции статей.**
   - Функция: `Retriever.retrieve(query, ctx, top_k=8)` — hybrid search (dense + sparse + RRF)
   - Фильтрация: по score_threshold для отсечения нерелевантных чанков
   - Логирование: «ШАГ 6. RAG вернул %d сниппетов — УСПЕХ»

7. **ШАГ 7. Финальный LLM-ответ с цитированием.**
   - Функция: `ContextBuilder.build_input()` с RAG-сниппетами
   - Системный промпт: «Ответь на вопрос, используя ТОЛЬКО предоставленные научные источники. Цитируй в формате [Author, Year]»
   - Функция: `stream_llm_to_stdout()` или `query_llm_simple()`
   - Логирование: «ШАГ 7. Финальный ответ с %d источниками — УСПЕХ»

8. **ШАГ 8. Возврат результата.**
   - Структура: `{answer, sources: [{paper_id, title, authors, year, citation_count, relevance_score}], collection_name, stats}`
   - Логирование: «ШАГ 8. Deep Academic Research завершён — УСПЕХ»

**Оптимизации:**
- **Дедупликация статей** по `paperId` между запросами
- **LLM-субагент** с минимальным контекстом (без истории) — дёшево и быстро
- **Временная коллекция** — изолирует результаты от основной базы знаний
- **Параллельная загрузка** статей через `asyncio.gather()` с семафором

   - Логирование: «ШАГ 2. S2 вернул %d рекомендаций — УСПЕХ»

3. **ШАГ 3. LLM-фильтрация и ранжирование.**
   - Функция: `OpenAIClient.complete()` — субагент без истории
   - Промпт: «Из %d рекомендованных статей выбери %d наиболее ценных. Для каждой дай краткую аннотацию (2-3 предложения) и оценку релевантности (1-10). Верни JSON»
   - Входные данные: title + abstract (200 символов) для каждой рекомендации
   - Парсинг: `StrictOutputParser.parse_json()`
   - Логирование: «ШАГ 3. LLM отобрал %d из %d рекомендаций — УСПЕХ»

4. **ШАГ 4. Обогащение аннотаций.**
   - Для отобранных статей: загрузка полных abstracts + tldr
   - Формирование развернутых карточек: title, authors, year, venue, citationCount, abstract, llm_annotation
   - Логирование: «ШАГ 4. Карточки %d статей обогащены — УСПЕХ»

5. **ШАГ 5. Финальный LLM-обзор.**
   - Системный промпт: «Сформируй обзор рекомендованных статей. Группируй по темам, укажи как они связаны с seed-статьями»
   - Функция: `stream_llm_to_stdout()` или `query_llm_simple()`
   - Логирование: «ШАГ 5. Финальный обзор рекомендаций — УСПЕХ»

6. **ШАГ 6. Возврат результата.**
   - Структура: `{seed_papers: [...], recommendations: [{paper_id, title, authors, year, citation_count, llm_annotation, relevance_score}], llm_overview, stats}`
   - Логирование: «ШАГ 6. Рекомендации статей завершены — УСПЕХ»

## UC-19: Автоматический научный литобзор (Literature Review via Semantic Scholar)

**Prerequisites:**
- **Payload:**
  ```json
  {
    "topic": "Применение графовых нейронных сетей в drug discovery",
    "search_queries_count": 5,
    "papers_per_query": 15,
    "max_total_papers": 50,
    "cluster_count": 4,
    "collection_name": "s2_litreview_<hash>",
    "output_language": "ru",
    "review_style": "structured"
  }
  ```
- **Внешние зависимости:** `pip install semanticscholar`
- **API:** `/graph/v1/paper/search`, `/graph/v1/paper/{paper_id}`
- **Конфигурация:** `CLOUDRU_API_KEY`, `QDRANT_URL` в .env; `S2_API_KEY` (опционально)
- **Требования:** `CloudRuEmbeddingClient` для dense эмбеддингов, `QdrantVectorStore` для хранения и кластеризации

**Actor:** Исследователь / PhD-студент / Аналитик

**Цель:** Автоматическое формирование структурированного литературного обзора по теме: multi-query поиск → сбор 50+ статей → кластеризация по семантической близости → Map-Reduce обзор по кластерам → финальная сборка.

**Шаги реализации:**

1. **ШАГ 1. Генерация разнообразных поисковых запросов.**
   - Функция: `OpenAIClient.complete()` с промптом: «Сгенерируй %d разнообразных поисковых запросов на английском для литобзора по теме: [topic]. Охвати разные аспекты: теория, применения, методы, датасеты, benchmarks»
   - Парсинг: JSON-массив строк
   - Логирование: «ШАГ 1. Сгенерировано %d поисковых запросов — УСПЕХ»

1b. **ШАГ 1b. LLM-инференс fieldsOfStudy.**
   - Функция: `S2Client.infer_fields(topic, ctx)` — автоматическое определение научных областей
   - Формирует `S2SearchFilter` для фильтрации
   - Логирование: «ШАГ 1b. Инференс fieldsOfStudy: [...] — УСПЕХ»

2. **ШАГ 2. Массовый поиск статей в Semantic Scholar.**
   - Функция: `S2Client.search_papers(query, limit=15, filters=S2SearchFilter)` × N запросов
   - Поля: `paperId, title, abstract, year, citationCount, authors, venue, fieldsOfStudy, tldr`
   - Дедупликация по `paperId`, ограничение `max_total_papers`
   - Приоритет: статьи с abstract, сортировка по citationCount
   - Логирование: «ШАГ 2. Собрано %d уникальных статей из %d запросов — УСПЕХ»

3. **ШАГ 3. Эмбеддинг + индексация всех статей.**
   - Текст для эмбеддинга: `S2Client.paper_embed_text(paper)` — title + abstract
   - Функция: `Chunker.split()` → индексация в коллекцию `s2_litreview_<hash>`
   - Метаданные в payload: `paper_id, title, year, citation_count, authors, venue`
   - Логирование: «ШАГ 3. Индексировано %d статей в коллекцию '%s' — УСПЕХ»

4. **ШАГ 4. Кластеризация статей через Qdrant.**
   - Для каждой статьи: dense-эмбеддинг уже в коллекции
   - Метод: k-means на основе dense-векторов (k=`cluster_count`)
   - Альтернатива: поиск ближайших к N центроидам (генерируем центроидные запросы через LLM)
   - LLM генерирует `cluster_count` подтем, для каждой → RAG-поиск → группировка
   - Логирование: «ШАГ 4. Статьи распределены по %d кластерам — УСПЕХ»

5. **ШАГ 5. Map-фаза: LLM-обзор каждого кластера.**
   - Для каждого кластера: top-K статей (title + abstract) → LLM-обзор
   - Промпт: «Напиши секцию литературного обзора по теме [cluster_label]. Используй предоставленные статьи, цитируй в формате [Author, Year]»
   - Ограничение конкурентности: `asyncio.Semaphore(2)` — не более 2 параллельных запросов к LLM
   - Логирование: «ШАГ 5. Обзор кластера %d/%d — УСПЕХ»

6. **ШАГ 6. Reduce-фаза: сборка финального литобзора.**
   - Объединение секций всех кластеров
   - Функция: `OpenAIClient.complete()` с промптом: «Собери из секций единый структурированный литобзор с введением, основной частью и заключением»
   - Добавление: библиография (список всех процитированных статей)
   - Логирование: «ШАГ 6. Финальный литобзор собран — УСПЕХ»

7. **ШАГ 7. Возврат результата.**
   - Структура: `{topic, review_text, sections: [{cluster_label, summary, papers: [...]}], bibliography: [...], collection_name, stats: {total_papers, clusters, total_tokens}}`
   - Логирование: «ШАГ 7. Литобзор завершён — УСПЕХ»

**Оптимизации:**
- **Multi-query поиск:** 5 разных запросов → покрытие разных аспектов темы
- **Дедупликация:** по `paperId` исключает повторы между запросами
- **Кластеризация через LLM:** генерация подтем + RAG-поиск вместо классического k-means → устойчивее и интерпретируемее
- **Map-Reduce:** параллельная обработка кластеров → линейное масштабирование
- **Временная коллекция:** результаты изолированы, можно переиспользовать или удалить

## UC-20: Безопасная транскрибация крупных аудио (Cloud.ru ASR)

**Prerequisites:**
- **Payload:** `{ "files": ["/abs/path/file1.ogg", "file2.m4a"], "output": "data/transcript/out.md" }`
- **Лимит Cloud.ru:** эффективный порог полезной нагрузки ~22–23 МБ (учитываем multipart overhead)
- **Форматы входа:** OGG/Opus, M4A/AAC, MP3; WAV/PCM нужно сжимать
- **Конфигурация:** `CLOUDRU_API_KEY` в .env; опционально `CLOUDRU_BASE_URL`, `ASR_MODEL`

**Actor:** Аналитик / Оператор / Фоновый джоб

**Цель:** Транскрибировать большие аудио без 413, с автоматической нарезкой, сжатием и сохранением порядка частей.

**Шаги реализации:**

1. **ШАГ 1. Валидация и определение формата.**
   - Проверка существования файлов, чтение размера, попытка определить кодек (bitrate, duration)
   - Лог: «ШАГ 1. Файл X: формат=..., размер=..., длительность=... — УСПЕХ/ОШИБКА»

2. **ШАГ 2. Проверка порога и выбор стратегии.**
   - Если формат поддержан и размер < 22 МБ → отправляем как есть
   - Если формат поддержан, но размер ≥ 22 МБ → перейти к сжатию/нарезке
   - Если формат не поддержан → перекодировать в целевой формат (Opus/AAC) с низким битрейтом
   - Лог: «ШАГ 2. Размер X > лимита 22 МБ — включаем сжатие/нарезку»

3. **ШАГ 3. Сжатие без раздувания.**
   - Целевой кодек: Opus 32–48 kbps mono 16 kHz (libopus) или AAC 48–64 kbps mono 16 kHz
   - После сжатия повторно проверяем размер; если всё ещё ≥ 22 МБ — нарезаем
   - Лог: «ШАГ 3. После сжатия размер=Y байт (лимит=22 МБ) — ПРОДОЛЖАЕМ/ПРЕРЫВАЕМ»

4. **ШАГ 4. Нарезка на чанки по времени.**
   - Делим на отрезки 2–5 минут так, чтобы каждый чанк после сжатия < 22 МБ
   - Сохраняем порядковые индексы для последующей сборки текста
   - Лог: «ШАГ 4. Файл X нарезан на N чанков (<=22 МБ каждый)»

5. **ШАГ 5. Последовательная очередь запросов к ASR.**
   - Semaphore = 1, отправляем чанки по порядку с request_id вида `file-i/chunk-j`
   - При 413 на чанке: немедленно логируем и прерываем обработку файла
   - Лог: «ШАГ 5. Чанк j/N размер=Y — ОТПРАВЛЯЕМ/ОШИБКА 413/УСПЕХ»

6. **ШАГ 6. Сборка транскрипта.**
   - Склеиваем тексты чанков в исходном порядке, добавляем разделители/таймкоды
   - Лог: «ШАГ 6. Собран транскрипт файла X: длина Z символов — УСПЕХ/ЧАСТИЧНО»

7. **ШАГ 7. Markdown-отчёт.**
   - Формируем общий отчёт: список файлов, размер до/после, число чанков, статусы
   - Лог: «ШАГ 7. Отчёт сохранён: path — УСПЕХ»

**Особенности:**
- Не использовать PCM (pcm_s16le) в fallback — он раздувает размер и ведёт к 413
- Для больших исходников сначала сжатие, потом нарезка; повторная отправка без уменьшения размера запрещена
- Конкурентность: только последовательная очередь для одного файла; разные файлы можно ставить в отдельные задачи при необходимости
- Лимит можно параметризовать (например, `ASR_MAX_BYTES=23000000`) для тонкой настройки под Cloud.ru

## Опущенные методы и причины

Следующие методы присутствуют в коде, но не вынесены в отдельные Use Cases:

### `SparseEmbeddingClient.embed_query()` / `embed_documents()`
- **Причина:** Уже покрыты UC-1 и UC-6 через `Retriever.retrieve()`. Retriever внутри вызывает эти методы — для пользователя интерфейс это `Retriever.retrieve()` с параметром `sparse_client`.

### `QdrantVectorStore.count_points(collection, filters)`
- **Причина:** Используется только внутри UC-12 (администрирование) для health check. Не выносится отдельно, т.к. это внутренний инструмент мониторинга.

### `ingest_and_index()` — standalone функция
- **Причина:** Это фасад/обёртка, объединяющая `DocumentIngestor.ingest()` + `Chunker.split()` + `EmbeddingClient.embed_texts()` + `VectorStore.upsert()`. Покрыта UC-6 (шаги 2-5), поэтому отдельный UC не требуется.

### `ToolRegistry.get_spec()` / `get_handler()`
- **Причина:** Внутренние методы для `ToolExecutor`, не предназначены для прямого использования в бизнес-логике.

## UC-хелперы в llm_service.py (секции 16, 16b)

При рефакторинге UC-скриптов повторяющийся код вынесен в
`llm_service.py` как набор хелперов. Это позволяет UC-скриптам оставаться
**тонкими**, фокусируясь только на оркестрации шагов.

### Фабрики и конфигурация

| Функция | Назначение |
|---------|-----------|
| `load_env_and_validate(env_dir, required_keys)` | Загрузка `.env` + валидация обязательных ключей. Возвращает `Dict` со всеми стандартными Cloud.ru переменными с дефолтами. Бросает `ValueError` при отсутствии ключа. |
| `create_rag_clients_from_env(collection, sparse_cache_dir)` | Фабрика RAG-клиентов: `embedding`, `sparse`, `vector_store`, `retriever` — всё из переменных окружения. |
| `create_cloudru_openai_client_from_env()` | Фабрика OpenAI-совместимого LLM-клиента из env. |
| `create_default_orchestrator_from_env(collection_name, enable_sparse)` | Полный `ChatOrchestrator` из env (LLM + RAG + Crawler + Tools). |

### Промпт-шаблоны

| Функция | Назначение |
|---------|-----------|
| `build_rag_prompt(query, snippets)` | Формирует промпт для RAG-ответа с инструкциями и списком источников. Используется в UC-2. |
| `build_web_prompt(query, snippets, chunks)` | Формирует промпт для ответа на основе веб-контента. Используется в UC-3. |

### LLM-обёртки

| Функция | Назначение |
|---------|-----------|
| `stream_llm_to_stdout(llm_client, prompt, ctx, ...)` | Стриминг ответа LLM с выводом токенов в stdout. Общий хелпер для UC-2, UC-3. |
| `query_llm_simple(llm_client, prompt, ctx, ...)` | Non-streaming запрос к LLM, возвращает текст ответа. Используется в UC-3. |

### Semantic Scholar классы (секция 16b)

| Класс/Константа | Назначение |
|---------|----------|
| `S2SearchFilter` | Dataclass-фильтр для поиска: `fields_of_study`, `year`, `min_citation_count`, `open_access_pdf`. Метод `to_api_params()` конвертирует в kwargs для S2 SDK. |

Статические: `paper_to_text`, `paper_embed_text`, `paper_citation_key`. |
| `S2_VALID_FIELDS_OF_STUDY` | Список 22 валидных категорий S2 (Computer Science, Medicine, ...). |
| `S2_PAPER_FIELDS` | Стандартный набор полей для запросов статей. |
| `S2_RATE_LIMIT_DELAY` | Задержка между запросами (0.6с). |

## Общие замечания по всем Use Cases

**Все UC используют общие компоненты:**

- `ChatOrchestrator` — центральный оркестратор всех операций
- `ContextBuilder` — сборка промптов из различных источников
- `ToolRegistry` + `ToolExecutor` — управление инструментами
- `QdrantVectorStore` — векторное хранилище с hybrid search
- `CloudRuEmbeddingClient` / `SparseEmbeddingClient` — генерация эмбеддингов
- `StrictOutputParser` + `JsonRepairLLM` — валидация и ремонт JSON
- `DocumentIngestor` + `Chunker` — обработка документов
- `OpenAIClient` + `Chunker` — Map-Reduce суммирование документов (UC-4)
- `AsyncTaskQueue` + `AsyncTask` + `WebhookClient` — асинхронная обработка с callback (UC-7, реализовано локально в UC_7.py)
- `ASRClient` + `asyncio.Semaphore` — пакетная транскрипция аудио (UC-11, реализовано локально в UC_11.py)
- `S2Client` + `S2SearchFilter` + `S2FieldInference` (pip: semanticscholar) — асинхронный клиент Semantic Scholar с фильтрацией и LLM-инференсом fieldsOfStudy (UC-15..UC-19)

**UC-хелперы (секции 16, 16b в llm_service.py):**
- `load_env_and_validate` — единая загрузка конфигурации для всех UC
- `create_rag_clients_from_env` — фабрика RAG-клиентов
- `build_rag_prompt` / `build_web_prompt` — шаблоны промптов
- `stream_llm_to_stdout` / `query_llm_simple` — LLM-обёртки
- `S2Client` / `S2SearchFilter` / `S2FieldInference` — Semantic Scholar клиент с фильтрацией

**Логирование во всех UC:**
- Формат: «ШАГ N. Описание — СТАТУС»
- Статусы: ОТПРАВЛЯЕМ, ОЖИДАЕМ, УСПЕХ, ОШИБКА, РЕМОНТ
- Все ключевые операции логируются для отладки и мониторинга
